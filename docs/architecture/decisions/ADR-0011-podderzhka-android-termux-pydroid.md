# ADR-0011: Поддержка запуска CLI на Android (Termux/Pydroid)

- **Статус:** принято
- **Дата:** 2026-05-19

## Контекст

catalib проектировался под запуск с ПК: деплой и диагностика завязаны на
`adb` (ADR-0004), `watch` — на `watchfiles` (Rust-бэкенд, ADR-0005),
`logs` — на `adb logcat`. Запрос: разрабатывать и деплоить плагины прямо
на телефоне, где запущен exteraGram, из Termux или Pydroid 3 — полный
паритет всех команд.

Ограничения окружений (проверено, см. ниже):

- На самом устройстве `adb` нет и не нужен: dev server exteraGram слушает
  `127.0.0.1:42690`, и процесс на том же устройстве подключается к нему
  напрямую — `adb forward` не требуется (он нужен только чтобы пробросить
  порт **с** устройства на ПК).
- `watchfiles` не имеет колёс под Android (Bionic libc, aarch64), сборка
  из исходников требует Rust, который на Termux проблемен (rustup не
  поддерживает host-triple `aarch64-linux-android`). Значит `watch` не
  может жёстко зависеть от `watchfiles`.
- Чтение **чужого** (exteraGram) logcat на Android требует разрешения
  `READ_LOGS` (root, adb-grant или Shizuku); обычное приложение
  Termux/Pydroid видит только свои логи.
- Termux — полноценный Linux-userland (есть `subprocess`, системные
  бинарники `logcat`/`getprop` в `/system/bin`). Pydroid 3 — песочница
  приложения: pip без компиляторов (только чистый Python), `subprocess`
  может быть урезан. Поведение должно деградировать, а не падать.
- Детект Android: Python 3.13+ (в т. ч. Termux) — `sys.platform ==
  "android"`; старый Python (Pydroid часто 3.11/3.9) — `"linux"`.
  Поэтому детект слоёный: `sys.platform` **или** переменные окружения
  (`ANDROID_ROOT`+`ANDROID_DATA`) **или** наличие `/system`.

## Решение

1. **`catalib.platforms`** — единый модуль детекта среды:
   `is_android()`, `android_flavor()` (`termux`/`pydroid`/`android`),
   `should_use_adb(explicit)`. Чистый, тестируется подменой
   `sys.platform`/окружения. Все команды спрашивают его, а не дублируют
   эвристику.
2. **Деплой без `adb` на устройстве.** `deploy_plugin` получает
   `use_adb: bool | None` (None = авто: на Android — без adb). Без adb:
   пропускается `adb forward`/`--remove`, подключение прямо к
   `127.0.0.1:<port>` (по умолчанию 42690 = порт dev server). Путь с
   внедрённым клиентом (тесты) не меняется. CLI получает тристейт
   `--adb/--no-adb` (по умолчанию авто).
3. **`watch` без Rust.** Новый `catalib.watching.iter_changes` выбирает
   бэкенд: `watchfiles`, если импортируется, иначе stdlib-поллинг
   (скан `mtime`/размера, интервал по умолчанию 1 с). `watch` больше не
   падает без `watchfiles` — это **ревизия ADR-0005**: пакет остаётся
   опциональным и предпочтительным (мгновенный, дешевле по CPU), но
   отсутствие даёт деградацию, а не ошибку.
4. **`doctor` под Android.** На устройстве проверки `adb`/«устройство»
   заменяются на: проверку среды (Android + flavor, информативно) и
   доступность dev server **напрямую** (`127.0.0.1:<port>`, без adb).
   `fail` по-прежнему только при старом Python / битом манифесте.
5. **`logs` под Android.** На устройстве вызывается системный `logcat`
   напрямую (без префикса `adb`). При отказе (нет `READ_LOGS`, urезан
   `subprocess`) — понятное сообщение и подсказки (Shizuku/adb-grant/
   root), корректный код возврата. Чистая фильтрация
   (`catalib.devicelogs.filter_log`) не меняется.

Команды `build`/`init`/`stubs`/`version` уже не зависят от устройства и
работают на Termux/Pydroid без изменений (только чистый Python +
файловая система + `importlib.resources`).

## Альтернативы

- **Свой нативный watcher (inotify через ctypes).** Отвергнуто:
  сложность и хрупкость ради того, что решает stdlib-поллинг; на
  ограниченных средах inotify тоже не гарантирован.
- **Требовать `adb` всегда (через Termux-`adb` по сети).** Отвергнуто:
  усложняет типовой сценарий «всё на телефоне», где adb лишний.
- **Свой Java-мост к logcat (READ_LOGS внутри приложения).** Отвергнуто:
  Termux/Pydroid — чужие приложения без этого разрешения; мост невозможен
  без root/Shizuku, а это вне зоны catalib (инструмент, не приложение).
- **Полный отказ `logs` на устройстве.** Отвергнуто пользователем:
  пытаться и честно подсказывать при отказе полезнее.

## Последствия

- Все 7 команд работают на Termux; на Pydroid — с деградацией там, где
  `subprocess` урезан (понятные сообщения вместо падений).
- `watch` приобретает универсальный поллинг-фолбэк (полезен и в CI).
- Новый публичный контракт: `catalib.platforms`, флаги `--adb/--no-adb`.
- Всё тестируется офлайн подменой среды/импорта (без устройства).
