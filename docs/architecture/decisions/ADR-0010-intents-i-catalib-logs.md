# ADR-0010: `intents` не оборачиваем; диагностика логами `catalib logs`

- **Статус:** принято
- **Дата:** 2026-05-18

## Контекст

Анализ `exteraGram-beta-universal-20260515.apk` для поиска пробелов
комфорта (см. блоки A–C) оставил один вопрос: модуль SDK `intents`,
ранее помеченный как «проверить перед обёрткой». Параллельно не хватало
быстрого способа посмотреть, что плагин пишет в лог на устройстве, не
открывая отдельный инструмент.

Проверка по APK и справочникам:

- В APK **нет** статического файла модуля `intents` (ни `.py`, ни
  `.pyi`, ни `.so`). `assets/chaquopy/app.imy` — **пустой** ZIP;
  `requirements-common.imy` не содержит `.py`. Python-модули SDK
  (`base_plugin`, `client_utils`, `intents`, …) предоставляются
  приложением в рантайме (Java/native через Chaquopy), а не лежат
  файлами, которые можно интроспектировать.
- `intents` **отсутствует** в выверенном справочнике SDK
  (`mcp/plan/sdk-reference/`) и в документации публичной поверхности.
- Поверхность `intents` не зафиксирована и может меняться между сборками
  exteraGram.

**Эмпирическая проверка (2026-05-18, одноразовый плагин-зонд через dev
server на живом устройстве).** Подтверждено: модуль — компилированный
`…/chaquopy/plugins-sdk/intents.so`, `__doc__ = None`, `__all__` нет.
Из 34 имён в `dir()` 33 — реэкспорт: `Intent`/`Uri` это
`android.content.Intent`/`android.net.Uri`; Java-классы через `jclass`
(`ArrayList`, `String`, `PyObject`, …); внутренняя машинерия байткода
(`BinaryOp`, `Comparison`, `Modifier`, `TypeId`, `Label`); утянутые в
неймспейс импорты (`dataclass`, `inspect`, `re`, `java`, `parse_qsl`,
`suppress`, `jclass*`, `log`, …). Собственный API — **один класс
`IntentsManager`** (`new_global_before_handler`,
`new_global_after_handler`, `parse`, `unhandle`; вложенные
`HandlerHandle`, `HandlerInfo`, `HandlerNotRegistered`) — глобальный
перехват Android-интентов. Сигнатуры методов не интроспектируются
(`.so`), документации нет. Вывод проверки совпал с гипотезой и усилил
решение.

## Решение

1. **`intents` не оборачивается** в `catalib.support`. Обёртка по
   догадке нарушила бы принцип ADR-0006/0007 (оборачиваем только
   **публичную, документированную** поверхность с офлайн-тестируемостью)
   и дала бы хрупкий контракт без возможности правдиво смоделировать
   заглушку. Прямой импорт на устройстве и так доступен — слой ничего не
   запрещает; в книге описан безопасный escape-hatch с оговоркой о
   нестабильности (раздел «Доступ к SDK»).
2. **Добавить `catalib logs`** — диагностика на устройстве: чтение
   logcat через `adb` с фильтром по `plugin_id` из манифеста.
   Поведение и команда (`logcat -d -t N`, опционально `logcat -c`,
   регистронезависимая подстрока, «хвост» при отсутствии совпадений)
   совпадают с инструментом MCP `adb_get_logs` — единый опыт.

Чистая фильтрация вынесена в `catalib.devicelogs.filter_log`
(тестируется офлайн), чтение logcat — в `catalib.deploy.adb.logcat`
(переиспользует существующую обвязку `adb`).

## Альтернативы

- **Тонкая обёртка `catalib.support.intents`.** Отвергнуто и после
  эмпирической проверки: реальный API — один недокументированный
  компилированный класс `IntentsManager` (узкая ниша — перехват
  интентов), сигнатуры не интроспектируются, контракт между сборками не
  гарантирован. Обёртка без офлайн-заглушки и без документации
  противоречит ADR-0006/0007 и быстро устареет; ценность ниже риска.
- **`catalib logs` через dev server.** Отвергнуто: протокол dev server
  (ADR-0004) не отдаёт логи; логи плагина идут в Android logcat. `adb`
  уже используется (ADR-0004) — переиспользуем.
- **Свой формат фильтра.** Отвергнуто: рассинхрон с MCP `adb_get_logs`,
  которым пользуются те же разработчики.

## Последствия

- `intents` остаётся доступен только прямым импортом на устройстве (с
  оговоркой); catalib не несёт за него API-ответственности.
- `catalib logs` — новый публичный контракт CLI; не требует устройства
  для офлайн-тестов (логика фильтра чистая, `logcat` подменяется).
- Зависимость от `adb` для `logs` — как у `watch --deploy`/`doctor`
  (опциональна, только для устройства).

## Связи

- Зависит от: ADR-0004 (adb/dev server), ADR-0006/0007 (паритет SDK).
- Затрагивает: `catalib/devicelogs.py`, `cli/logs_command.py`,
  `deploy/adb.py` (`logcat`).
- Документация: книга — «Доступ к SDK» (intents), CLI «catalib logs».
